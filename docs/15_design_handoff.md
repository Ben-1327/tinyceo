# 15 Design Handoff (UI/UX Only)

このドキュメントは、デザイン担当に渡すための専用仕様。
目的は「ゲーム仕様を変えずに、UI/UX/ビジュアルだけを高品質に設計する」こと。

## 1. デザイン担当の責務（担当範囲）
- 情報設計（画面構造、視線誘導、優先順位）
- UI設計（レイアウト、コンポーネント、状態別表示）
- ビジュアル設計（カラー、タイポ、アイコン、イラスト方針）
- モーション設計（最小限の遷移、通知の見せ方）
- コピー設計（短い文言、状態説明、プライバシー文言）
- アセット仕様化（必要な画像/アイコンと書き出し条件）

## 2. デザイン担当の非責務（変更してはいけない範囲）
- ゲームロジック（経済、カード効果、進行条件）
- `data/*.json` の数値、カード内容、バランス
- シミュレーション処理順と時間モデル
- プライバシー原則（内容収集しない、ローカル完結）

## 3. 固定制約（必ず守る）
- メニューバー常駐前提（ポップオーバー中心）
- Low-interruption（1操作 60〜120秒）
- 通知は最小限（通常はカード到着と重大Crisisのみ）
- v0.1カードは3択固定
- 作業内容を覗いている印象を与えるUIは禁止

## 4. 参照元（デザイン担当が読むべき既存仕様）
- 体験原則: `docs/00_overview.md`
- コアループ/時間: `docs/02_core_loop_and_time_model.md`
- UX/通知: `docs/10_ux_and_notifications.md`
- アート方針: `docs/11_art_audio_and_assets.md`
- プライバシー文言: `docs/12_privacy_and_telemetry.md`
- 実装固定ルール: `docs/14_implementation_decisions.md`

## 5. 必須画面・状態
- メニューバーアイコン（通常 / 注意 / 危険 / 未処理カードあり）
- ポップオーバー Home（主要指標 + 進行中プロジェクト + Inbox件数）
- カード一覧（Inbox）
- カード詳細（タイトル、背景説明、3択、選択結果）
- 設定（モード切替、通知ON/OFF、作業連携ON/OFF）
- 初回オンボーディング（何を収集し、何を収集しないか）

## 6. コンポーネント定義が必要な要素
- KPIカード（Cash / Runway / Reputation / TeamHealth / TechDebt）
- 進捗バー（プロジェクト進行）
- リスクバッジ（危険状態の視覚化）
- 選択肢ボタン（メリット/デメリットが読み取りやすいこと）
- 軽量通知トースト（結果表示）

## 7. 状態設計（最低限）
- 通常運転
- 資金繰り悪化
- TechDebt悪化
- TeamHealth悪化
- Crisis発生中
- Inbox満杯

## 8. コピー指針
- 断定的で短い文を使う（長文説明を避ける）
- 効果説明は「何が上がる/下がる」を即読できる形にする
- プライバシー文言は以下をほぼそのまま使う
  - 「あなたの作業内容は記録しません。アプリ種別の滞在時間をカテゴリに集計するだけです。」
  - 「このデータはローカルに保存され、外部に送信されません。」

## 9. 納品物（デザイン担当が出すもの）
- 画面フロー図（初回起動〜通常運用〜危機対応）
- 主要画面の高解像度モック（通常/危機状態を含む）
- コンポーネント仕様（状態、余白、フォント、色、最小サイズ）
- デザイントークン（Color / Type / Spacing / Radius / Shadow）
- アイコン/イラストの使用ガイド
- エンジニア向け実装注記（各UIの意図、禁止事項）

## 10. 実装受け入れ条件（デザイン観点）
- 10分使っても通知が過剰に感じない
- KPIと危険状態が2秒以内に認識できる
- カード処理が1分前後で完了できる
- プライバシー不安を生む文言・演出がない
- 小ウィンドウでも可読性が崩れない
